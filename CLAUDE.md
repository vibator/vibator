# vibator — agent guide

Read this fully before writing code here.

## What this is

An **AI-first quality gate engine**. Configurable, glob-scoped rules that report
what is wrong, what was expected, and the concrete fix — in three separate
fields, so a human reads a sentence and an agent can act on `fix` without
parsing prose.

It exists because linters check syntax and type checkers check types, and
neither catches what generated code actually gets wrong: files that grow without
limit, a translation key seeded to one locale, an env var read but never
documented, a migration never regenerated, a deprecated API that still compiles.

Published to npm as `vibator`. **Nothing in this repository may name a
downstream project.** It was extracted from one, and staying generic is the
whole point.

## Using it

```sh
vibator                      # run every enabled rule
vibator --only max-lines     # comma-separated rule ids
vibator --config path.json   # explicit config file
vibator --reporter json      # machine-readable, for CI and agents
vibator --staged             # judge only files staged for the next commit
vibator --changed            # judge only uncommitted changes
vibator --since origin/main  # judge only files changed since a ref
vibator list                 # every rule, its default severity and title
vibator explain <rule>       # the guideline in force for a rule
vibator docs <topic>         # print a bundled doc (writing-rules, configuration)
vibator init                 # write a starter vibator.json
vibator skills --install     # copy bundled agent skills into .claude/skills/
```

Exit code 1 on any `error`-severity finding, or if a rule crashes. Warnings
alone exit 0.

Configured by `vibator.json` at the project root — see
[docs/configuration.md](./docs/configuration.md).

## Layout

```
src/
  core/        engine, config, discovery, context, plugins, diagnostics
  reporters/   pretty (TTY, progress) and json
  rules/       one file per built-in rule
  cli.ts       command line entry point
  cli/         argument parsing and the informational subcommands
  schema.ts    emits schema.json from the zod schemas
  catalog.ts   emits docs/rule-catalog.md from the same schemas
docs/
  configuration.md   config + CLI reference
  writing-rules.md   the full authoring guide
  rule-catalog.md    every rule + options, GENERATED by `npm run generate`
  rules/*.md         one guideline per rule, served by `explain`
skills/        Claude Code skills, shipped and installable
.github/       CI (quality.yml), releases (release.yml), dependabot
```

Releases are semantic-release over Conventional Commits (`.releaserc.json`),
cut manually via the Release workflow, published to npm. Commit types matter:
`fix:`/`feat:` release, and CI lints the messages.

## Working on it

```sh
npm run verify    # the whole gate: lint, arch, knip, build, test, dogfood
npm run build     # tsc + regenerate schema.json and docs/rule-catalog.md
npm test          # vitest
npx vibator       # dogfood: this repo lints itself via vibator.json
npm run lint      # Biome — strict format + lint (complexity ≤ 8, fn ≤ 25 lines)
npm run arch      # dependency-cruiser — layer boundaries, no static `typescript`
npm run knip      # dead code and unused dependencies
```

`npm run build` regenerates `schema.json` and `docs/rule-catalog.md` from the
rules' own zod schemas, so editor autocomplete and the options catalog cannot
drift from what the tool accepts. Run it after touching any rule's options and
commit the regenerated files — CI checks they are in sync.

## Design invariants

Do not undo these without a deliberate decision — each was arrived at by hitting
the failure it prevents.

- **No baselines, no suppression files.** A gate that lets you record violations
  and move on stops being a gate. The only escape is `// vibator-ignore:
  <reason>` on the line above, and the reason is mandatory — the bare marker
  must not match.
- **Three diagnostic fields stay separate.** `message` (what is wrong),
  `expected` (the standard, positively stated), `fix` (the next action). Never
  collapse them into one sentence; the JSON consumer is the point.
- **Every rule runs, even after one fails.** Stopping at the first failure hides
  everything behind whichever gate is ordered first.
- **Discovery defers to git** — tracked plus untracked-not-ignored. This is what
  keeps generated output and native build shells out of results. A plain
  filesystem walk reports files nobody wrote and nobody can fix.
- **Read through `context`, never `fs`.** Reads and TypeScript programs are
  memoized for the run; several rules sharing a tsconfig cost one type-check
  between them, not one each.
- **`typescript` is an optional peer.** Load it with `await import`, never at
  module scope, so text-only users never install it.
- **Rules that cannot act without options default to `off`**, so a zero-config
  run in a fresh project works rather than throwing.
- **Incremental adoption is `--changed`/`--since`, never a baseline.** Scoping
  the run to touched files gates new work without recording old debt as
  acceptable.
- **JSON output is self-sufficient for agents.** Findings carry `snippet` and
  `docs[].absolutePath`, so a consumer never needs to know where the package
  manager put this package. Nothing in docs or skills may assume
  `node_modules/` layout — that is what `vibator docs <topic>` exists for.
- **Pattern-shaped standards go in `banned-patterns` options**, not new rules.
  A built-in earns its place only when a regex over lines cannot express it.

## Adding a rule

Use the `writing-vibator-rules` skill, or read
[docs/writing-rules.md](./docs/writing-rules.md). In short: a rule is a plain
object with an `id`, a guideline in `docs/rules/`, an options schema, glob
defaults, and either `checkFile` (per file, progress handled for you) or `check`
(per project). Register it in `src/rules/index.ts`.

A rule earns its place only if it is deterministic, actionable in one line, low
on false positives, and not already covered by a formatter or type checker.

## Two traps in the TypeScript rules

Both cost real debugging; both have regression tests in
`src/rules/deprecation-analysis.test.ts`.

- **A symbol merges the JSDoc of every overload.** Asking whether
  `querySelectorAll` is deprecated answers yes, on the strength of a deprecated
  tag-name overload nobody called. Judge calls by
  `checker.getResolvedSignature`, not by the symbol.
- **An imported symbol resolves to the local import binding**, whose declaration
  carries no tag. Always unwrap with `getAliasedSymbol` first — skipping this
  makes every deprecated import silently invisible.

## Known gaps on the way to production

- **No smoke tests against fixture projects.** Every first-run bug found so far
  (required options throwing on zero-config, a missing `.env.example` crashing a
  rule, `.jsx` parsed as TypeScript) only appeared in a project that was not the
  one it was developed in. A fixture suite running the CLI against a bare JS
  project, a CSS-only project and a TS monorepo would catch the next one.
- **A partly deprecated symbol reports its import line.** Symbol tags merge
  across overloads, so importing a function whose *any* overload is deprecated
  reports the import even when only current overloads are called. Recorded as a
  test; fixing it means requiring *every* declaration to be deprecated before
  reporting a bare reference.
- **`no-deprecated-apis` costs seconds** — roughly 2s building programs and 2s
  resolving. A content-hash result cache would make repeat runs near-free.
- **External tools are not wrapped.** A `command` rule type would let one runner
  drive a formatter, an import-boundary checker and a test suite under a single
  progress display.
- **CSS/SCSS have no parser.** File-level rules cover stylesheets already;
  anything structural would need a grammar, and probably should not live here.

## Conventions

- TypeScript strict, ESM, Node ≥22.
- TSDoc on every declaration, exported or not, with `@param` and `@returns`.
  The repo enforces this on itself via `tsdoc-coverage`.
- Names carry meaning; no `data`, `res`, `tmp`.
- Functions stay short. Files stay under 400 lines — including this package's
  own, which is why the TSDoc analysis is split across `src/rules/tsdoc/`.
