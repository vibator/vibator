# vibator agent guide

Read this fully before writing code here.

## What this is

A quality gate framework for coding agents: a runner for configurable,
glob-scoped rules. It targets JavaScript and TypeScript projects, though some
rules apply to any file their globs select. Every finding reports three
separate fields: `message` (what is wrong), `expected` (the standard), and
`fix` (the next action). Humans read them as a sentence; tools and agents act
on `fix` without parsing prose.

It exists because the standards an agent is given in a prompt stop being
applied as the context grows, and because linters check syntax and type
checkers check types. Neither catches a class of defects common in generated
code: files that grow without limit, a translation key added to one locale
only, an env var read but never documented, generated files never
regenerated, deprecated APIs that still compile. Without a runner, an agent
enforces those with throwaway scripts.

Published to npm as `vibator`. Nothing in this repository may name a
downstream project. It was extracted from one, and it must stay generic.

## Using it

```sh
vibator                      # run every enabled rule
vibator --only max-lines     # comma-separated rule ids
vibator --config path.json   # explicit config file
vibator --reporter json      # machine-readable, for CI and agents
vibator --staged             # check only files staged for the next commit
vibator --changed            # check only uncommitted changes
vibator --since origin/main  # check only files changed since a ref
vibator list                 # every rule, its default severity and title
vibator explain <rule>       # the guideline in force for a rule
vibator docs <topic>         # print a bundled doc (writing-rules, configuration, rules)
vibator init                 # write a starter vibator.json
vibator skills --install     # copy bundled agent skills into .claude/skills/
```

Exit code 1 on any error-severity finding, or if a rule crashes. Warnings
alone exit 0.

Configured by `vibator.json` at the project root. See
[docs/configuration.md](./docs/configuration.md).

## Layout

```
src/
  core/        engine, config (+ extends), discovery, context, plugins,
               diagnostics
  reporters/   pretty (TTY, progress) and json
  rules/       one file per built-in rule
  cli.ts       command line entry point
  cli/         argument parsing and the informational subcommands
  schema.ts    emits schema.json from the zod schemas
  catalog.ts   emits docs/rule-catalog.md from the same schemas
docs/
  configuration.md   config and CLI reference
  writing-rules.md   the full authoring guide
  rule-catalog.md    every rule and its options, GENERATED by `npm run generate`
  rules/*.md         one guideline per rule, served by `explain`
skills/        Claude Code skills, shipped and installable
.github/       CI (quality.yml), releases (release.yml), dependabot
```

Releases are semantic-release over Conventional Commits (`.releaserc.json`),
cut manually via the Release workflow, published to npm. Commit types
matter: `fix:` and `feat:` release, and CI lints the messages.

## Working on it

```sh
npm run verify    # the whole gate: lint, arch, knip, build, test, self-check
npm run build     # tsc + regenerate schema.json and docs/rule-catalog.md
npm test          # vitest
npx vibator       # the repo checks itself via vibator.json
npm run lint      # Biome: strict format + lint (complexity <= 8, fn <= 25 lines)
npm run arch      # dependency-cruiser: layer boundaries, no static `typescript`
npm run knip      # dead code and unused dependencies
```

`npm run build` regenerates `schema.json` and `docs/rule-catalog.md` from
the rules' own zod schemas, so editor autocomplete and the options catalog
cannot drift from what the tool accepts. Run it after touching any rule's
options and commit the regenerated files. CI checks they are in sync.

Tests: unit tests live next to their modules; `src/cli.e2e.test.ts` runs the
real CLI against throwaway fixture projects (bare JS, CSS-only, `init`).
Coverage thresholds live in `vitest.config.ts`; raise them when coverage
rises, never lower them to make a run pass.

## Design invariants

Do not undo these without a deliberate decision. Each one exists because the
failure it prevents actually happened.

- **No baselines, no suppression files.** Recorded violations stop being
  violations. The only escape is `// vibator-ignore: <reason>` on the line
  above; the reason is required and the bare marker must not match.
- **Three diagnostic fields stay separate.** `message`, `expected`, `fix`.
  Never collapse them into one sentence; JSON consumers depend on the split.
- **Every rule runs, even after one fails.** Stopping at the first failure
  hides everything behind whichever rule runs first.
- **Discovery defers to git**: tracked plus untracked-not-ignored. This
  keeps generated output and native build shells out of results. A plain
  filesystem walk reports files nobody wrote and nobody can fix.
- **Read through `context`, never `fs`.** Reads and TypeScript programs are
  memoized for the run; rules sharing a tsconfig cost one type check total.
- **`typescript` is an optional peer.** Load it with `await import`, never
  at module scope, so text-only users never have to install it.
- **Rules that cannot act without options default to `off`**, so a
  zero-config run in a fresh project works instead of throwing.
- **Incremental adoption is `--changed` and `--since`, never a baseline.**
  Scoping the run to touched files checks new work without recording old
  violations as accepted.
- **JSON output is self-sufficient.** Findings carry `snippet` and
  `docs[].absolutePath`, so a consumer never needs to know where the package
  manager installed this package. Nothing in docs or skills may assume a
  `node_modules/` layout; that is what `vibator docs <topic>` is for.
- **Pattern-shaped standards go in `banned-patterns` options**, not new
  rules. A built-in is justified only when a regex over lines cannot express
  the check.

## Adding a rule

Use the `writing-vibator-rules` skill, or read
[docs/writing-rules.md](./docs/writing-rules.md). In short: a rule is a
plain object with an `id`, a guideline in `docs/rules/`, an options schema,
glob defaults, and either `checkFile` (per file, progress handled for you)
or `check` (per project). Register it in `src/rules/index.ts`.

A rule qualifies only if it is deterministic, actionable in one line, low on
false positives, and not already covered by a formatter or type checker.

## Two pitfalls in the TypeScript rules

Both cost real debugging time. Both have regression tests in
`src/rules/deprecation-analysis.test.ts`.

- **A symbol merges the JSDoc of every overload.** Asking whether
  `querySelectorAll` is deprecated answers yes, because one overload nobody
  called is deprecated. Judge calls by `checker.getResolvedSignature`, not
  by the symbol.
- **An imported symbol resolves to the local import binding**, whose
  declaration carries no tag. Always unwrap with `getAliasedSymbol` first.
  Skipping this makes every deprecated import invisible.

## Known gaps

- **A partly deprecated symbol reports its import line.** Symbol tags merge
  across overloads, so importing a function with any deprecated overload
  reports the import even when only current overloads are called. Recorded
  as a test. Fixing it means requiring every declaration to be deprecated
  before reporting a bare reference.
- **`no-deprecated-apis` costs seconds**: roughly 2s building programs and
  2s resolving. A content-hash result cache would make repeat runs cheap.
- **External tools are not wrapped.** A `command` rule type would let one
  runner drive a formatter, an import-boundary checker and a test suite
  under a single progress display.
- **CSS/SCSS have no parser.** File-level rules cover stylesheets already.
  Anything structural would need a grammar and probably should not live
  here.

## Conventions

- TypeScript strict, ESM, Node 22 or later.
- TSDoc on every declaration, exported or not, with `@param` and `@returns`.
  The repo enforces this on itself via `tsdoc-coverage`.
- Names carry meaning; no `data`, `res`, `tmp`.
- Functions stay short. Files stay under 400 lines, including this
  package's own, which is why the TSDoc analysis is split across
  `src/rules/tsdoc/`.
- Documentation and user-visible strings use plain, direct language: no
  em-dashes, no aphorisms, no marketing phrasing. See CONTRIBUTING.md.
